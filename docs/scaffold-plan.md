# Stratezone Scaffold Plan

This document defines the first real project scaffold target for Stratezone.

Its job is to keep the first engine setup small, concrete, and useful for the first playable mission. It does not create the Godot project by itself; it tells future implementation work what the scaffold should contain and how to know when it is complete.

## Documentation Role

- **Doc role:** Active source of truth for the first Godot project scaffold.
- **Owns:** first scaffold layout, engine setup expectations, early validation targets, and scaffold exit criteria.
- **Does not own:** game vision, final balance, mission scripts, art direction, or long-term packaging policy.
- **Read when:** creating the Godot project, adding root scripts, wiring first validation, or deciding where new files belong.
- **Do not read for:** exact gameplay rules, final unit stats, or narrative design.

## Locked Prototype Stack

The first prototype stack is locked as:

- **Engine:** Godot 4
- **Language:** C#
- **Platform target:** Windows desktop first
- **Distribution direction:** packaged desktop builds suitable for itch.io or Steam later
- **Visual mode:** readable 2D top-down

Do not add another engine, renderer, frontend stack, or large framework during the scaffold. If the project ever pivots away from this stack, update this file, `README.md`, `AGENTS.md`, and `docs/technical-architecture.md` in the same pass.

## Scaffold Goals

The scaffold should make it possible to start building the greybox mission without guessing about file ownership.

The first scaffold should provide:

- a Godot 4 C# project under `game/`
- a predictable place for scenes, scripts, data, and placeholder assets
- root-level commands or documented command paths for build and test checks
- enough C# structure to keep simulation rules separate from Godot presentation nodes
- a small smoke scene that proves the project opens and runs

It should not provide:

- final art
- final mission balance
- full UI polish
- multiplayer setup
- procedural generation
- campaign persistence
- store/export automation beyond a basic direction

## Minimum Godot Scaffold

The first Godot pass should include:

- `game/project.godot`
- one main scene that launches without errors
- one placeholder map scene or mission scene for First Landing
- C# project files generated by Godot
- a plain simulation namespace/folder that can compile without depending on scene-only truth
- a presentation namespace/folder for scene adapters, camera, selection visuals, and UI wiring
- placeholder shapes or simple sprites for early units/buildings

The first scene can be ugly. It should be useful.

## First Validation Targets

Current validation command:

- `python tools/validate_content.py`

Godot .NET 4.6.2 and .NET SDK 8 are the current local tool targets. If `godot` is not on PATH in the current shell, use the installed Godot console executable directly from the WinGet package folder.

Open/compile validation should use:

- Godot project load or headless sanity check
- C# build or compile check
- simulation unit tests
- content data validation once data files exist
- basic run/smoke check for the main scene

Prefer root-level commands if practical so future agents do not need to remember engine-specific paths.

## Scaffold Complete Means

The scaffold is complete when:

- the Godot 4 C# project opens from `game/`
- the main scene runs without errors
- C# compiles cleanly
- the repo has obvious places for simulation, presentation, content data, tests, and tools
- the first validation commands are documented in `docs/engineering-standards.md`
- no gameplay rule is forced to live only inside a visual scene script

Current status: the repo scaffold, content validation, C# compile check, and Godot headless smoke check pass on this machine. The first greybox shell spawns placeholder units from content data and supports camera pan/zoom, click selection, and right-click move.

At that point, implementation should move to the greybox First Landing loop, not more scaffold polishing.
